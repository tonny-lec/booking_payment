# セルフチェックリスト

タスク完了を宣言する前に、対応するフェーズのチェックを実行すること。
チェックが通らない場合は、次にやるべきことを宣言して止まる。

---

## Phase 0: Prime（着手前）

- [ ] 作業目的を1文で言える
- [ ] 成功条件（Acceptance Criteria）を言える
- [ ] 影響範囲（触る可能性のある場所）を列挙できる
- [ ] 検証方法（どうやって正しさを確認するか）を言える

---

## Phase 1: PRD（要件定義）

- [ ] ノンゴールが明確（スコープ膨張を止める栓がある）
- [ ] 受け入れ条件が「観測可能」（Yes/Noで判定できる）
- [ ] 未確定事項が列挙され、潰し方が書かれている

---

## Phase 2: Plan（実装計画）

- [ ] 実装ステップが順序つきで、各ステップが"小さい"
- [ ] 検証が具体（どのコマンド/どの観測でOKにするか）
- [ ] "やらないこと"がPlanにも反映されている
- [ ] 依存する外部要因（環境変数、APIキー等）が明記されている

---

## Phase 3: Context Reset（計画→実装の切替）

- [ ] 今から参照するのは「PLAN」だけ、と宣言した
- [ ] 追加で参照するdocがあるなら列挙した

---

## Phase 4: Implement（実装）

- [ ] Planのステップ番号に沿って進んでいる
- [ ] 変更理由を説明できる
- [ ] 余計なリファクタでスコープを増やしていない
- [ ] "推測"の部分が検証項目に入っている

---

## Phase 5: Validate（検証）

- [ ] PRDの受け入れ条件に対して検証結果を紐づけた
- [ ] 失敗時の切り分けログが残っている
- [ ] "通った/通ってない"が明確

---

## Phase 6: Report（実装レポート）

- [ ] 何を変えたか（概要）
- [ ] どこを変えたか（影響範囲）
- [ ] どう検証したか（結果つき）
- [ ] 残タスク/既知の制限（次にやること）

---

## タスク完了時（共通）

- [ ] **PRD整合性**: 実装したコードはPRD（North Star）の仕様と一致しているか？
- [ ] **コンテキスト汚染**: 実行フェーズで、計画策定時の古い会話履歴を引きずっていないか？
- [ ] **ルール遵守**: モジュラールール（API/UI等の個別規定）に従った実装になっているか？
- [ ] **システム進化**: 発生したエラーに対して、ルールファイルへのフィードバック（更新）を提案したか？
- [ ] **テスト**: 変更箇所に対するテストはパスしたか？

---

## RCA（Root Cause Analysis）作成時

RCAを作成する条件：
- 同種のミスが2回目
- テスト不足で見逃した
- 仕様の読み違い
- 依存関係/環境差分でハマった

RCA完了チェック：
- [ ] "人の注意力"に依存する対策で終わっていない
- [ ] ルール or テンプレ or 検証手順のどれかが更新される
