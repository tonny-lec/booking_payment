# Codex-first Workflow

## Default: Core Loop

短い修正、レビュー、デバッグ、説明、調査では、以下を標準とする。

### 1. Explore
- まず対象ファイル、設定、起動方法、テスト方法を読む
- 探索で解ける不確定は質問しない
- 読み過ぎず、今触る領域だけを特定する

### 2. Confirm Intent If Needed
- 仕様判断、優先順位、トレードオフが残る場合だけ確認する
- 実装に影響しない細部では止まらない

### 3. Edit or Analyze
- 実装タスクでは局所的に編集する
- レビュー、説明、調査では変更せずに分析を返す
- 無関係な整理や大規模な横断修正は混ぜない

### 4. Validate
- 変更対象に見合う検証を行う
- 可能ならテスト、lint、build、手動確認のいずれかを実施する
- レビューや調査では、根拠ファイルと観測事実を示す

### 5. Report
- 変更概要
- 影響範囲
- 検証結果
- 残課題または既知の制限

## Extended: Plan Flow

以下では、Core Loop ではなく明示的な Plan を作る。
- 複数ファイルにまたがる実装
- 長時間タスク
- 複数の判断点があるタスク
- 外部依存、データ移行、互換性考慮があるタスク
- 高リスク変更

### Phase 0: Prime
- 目的、成功条件、影響範囲、検証方法を固める

### Phase 1: PRD
- コード/インフラ変更では、承認済みPRDを確認する
- 実装しない調査やレビューでは不要

### Phase 2: Plan
- 変更方針、実装順序、検証手順、失敗時の戻し方を定める

### Phase 3: Reset
- 実装に不要な長文脈を捨てる
- `Plan` と必要最小限の参照文書だけを持ち込む

### Phase 4: Implement
- Plan に沿って小さく変更する

### Phase 5: Validate
- 受け入れ条件と検証結果を結びつける

### Phase 6: Report
- 復元に必要な要点だけ残す

---

## Task Types

### Implement
- 既存仕様または承認済みPRDに基づいて変更する

### Review
- バグ、回帰、リスク、テスト不足を優先して確認する

### Debug
- まず再現条件、失敗箇所、観測ログを固める

### Explain
- 変更せず、対象の振る舞いと意図を説明する

### Research
- 変更せず、現状、制約、選択肢を整理する

## Commandify and Skills
- 同じ手順を 2 回以上繰り返したら、テンプレート、コマンド、skill 化を検討する
- skill 化の対象は、再利用性が高く、入力と出力が安定し、評価しやすい作業とする
